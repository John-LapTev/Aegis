## 🔴 Важное (чинить в первую очередь)

### 1. [HIGH] Авто-обновление: при провале запуска ПОСЛЕ подмены `.exe` откат не восстанавливает — «кирпич»
`UpdateService.cs` (`DownloadAndApplyAsync` :96-154, `TryRollback` :157-175). Оба `File.Move` прошли (старый→`.old`, новый→на место), затем `Process.Start` упал (битый .exe / антивирус / отмена UAC). `TryRollback` восстанавливает только при `!File.Exists(exePath)` — но там уже лежит новый (битый) .exe → рабочий `.old` НЕ возвращается. Итог: `Aegis.exe` битый, `.old` рабочий, программа не стартует; спасение — вручную переименовать `.old`.
**Фикс:** восстанавливать ВСЕГДА, когда есть валидный `.old` (отодвинуть текущий exePath, `File.Move(oldPath, exePath)`), а не только при отсутствии exePath.

### 2. [HIGH] Авто-обновление: нет проверки целостности скачанного `.exe` (усугубляет #1)
`UpdateService.cs` `DownloadFileAsync` :177-201. Чистый обрыв соединения (EOF раньше времени) завершает цикл штатно на неполном файле; сверки `read == Content-Length`, размера, хэша нет. Усечённый `.exe` подменяет рабочий → сценарий #1 (кирпич). `total` уже читается для прогресса, но не используется для проверки.
**Фикс:** после скачивания проверять `total > 0 && read != total` → ошибка ДО подмены (пока рабочий .exe цел). Плюс класть в релиз SHA-256 и сверять (я публикую релизы). Тогда битая/подменённая загрузка проваливает обновление безопасно.

### 3. [HIGH] Авто-обновление: ошибка НЕ видна в главной плашке
`MainWindow.axaml` (плашка) + `MainWindowViewModel.ApplyUpdateAsync`. `UpdateStatus` выводится только внутри `IsVisible="{Binding IsUpdating}"`; при ошибке `finally` ставит `IsUpdating=false` → блок с ошибкой скрывается, плашка возвращается к «Нажмите Обновить…», `UpdateAvailable` остаётся true. Пользователь жмёт снова → та же тихая неудача. (В «О программе» ошибка видна, но новичок туда не пойдёт.)
**Фикс:** показывать `UpdateStatus` в плашке и при `!IsUpdating` (при непустом тексте), тревожным цветом.

### 4. [MEDIUM, безопасность] Кеш онлайн-репутации может перекрасить Danger-находку в зелёное
`OnlineReputationChecker.cs` :70-74 (чтение кеша) и :85-90 (запись). В кеш по пути кладётся уже «прогейченное по severity» значение (`Clean && Severity != Danger`); при кеш-хите severity повторно НЕ проверяется. Сценарий: один exe даёт две находки (процесс Danger + автозапуск Warning) с одинаковым путём. Если первой проверилась НЕ-Danger и чисто → кеш `true`; вторая (Danger) ловит хит → `IsVerifiedSafeOnline=true` → показывается зелёной. Прямо нарушает правило «Danger по чистому онлайн-вердикту в зелёное не переводить» (прячет опасное).
**Фикс:** кешировать ЧИСТЫЙ вердикт (`Verdict == Clean`), а гейт `&& Severity != Danger` применять пер-находка И на ветке кеш-хита.

## 🟡 Стоит поправить (MEDIUM)

- **ИИ-ссылки недоверенных доменов всё ещё кликабельны** (`FindingViewModel` LinkLabel/OnAiAnswerChanged): `TrustedDomains.IsTrusted` влияет только на ПОДПИСЬ, но кнопка рендерится всё равно. Фейковый/галлюцинированный URL «Открыть ссылку (drivers-xyz.com)» доверчивый новичок откроет. **Фикс:** недоверенные — не кнопкой (текст + пометка «сторонний сайт, осторожно») или предупреждение перед переходом.
- **Дедуп пакетов: нет кода-денилиста** (`DashboardViewModel`/`RedistDeletionMatcher`): безопасность на 100% на промпте. Если ИИ выдаст `DEL::Microsoft .NET Framework 4.8` или единственный VC++ линейки — матчер пропустит (единственный кандидат, разрядность сходится). **Фикс:** хардкод-денилист — никогда не отдавать `.NET Framework`, DirectX, WebView2 и единственный экземпляр линейки/разрядности VC++.
- **Веб-сниппеты в промпт без изоляции/лимита** (`WebAugmentedAiAssistant`): title/snippet/url вставляются как есть; в нише поиска драйверов — SEO-помойки с возможной инъекцией. **Фикс:** обрезать сниппеты (300-500 симв.), обрамлять «данные из сети, не инструкции», фильтровать итоговые ссылки через `TrustedDomains`.
- **Остатки: файлы удаляются насовсем + все галочки предвыбраны** (`LeftoverService`/`LeftoverConfirmWindow`): для «жмёт не думая» = один клик стирает навсегда; совпадение по имени может зацепить чужую папку. (Насовсем — решение Ивана 1204.) **Фикс:** не предвыбирать галочки у файлов/папок (или предвыбирать только реестр, он с бэкапом).
- **`FindingDataKeys` заведён, но миграция не доведена** (M1): новый/вынесенный код (`HealthTiles`, `FindingAiPrompt`, `DriversScanner`) ещё на магических строках (`healthIcon`/`placeholder`/`deviceId`/`bytes`/`publisher`). Довести producer↔consumer пары.
- **`FindingAiPrompt.WebQuery` читает несуществующие ключи `hwid`/`vid`** — мёртвая ветка (ни один сканер не пишет). Убрать.
- **`ConnectivityWatcher` не `IDisposable`** — таймер/`NetworkChange` не отписываются (латентно: один экземпляр, `Start` однажды). Сделать освобождаемым.
- **UpdateNotes качаются, но нигде не показаны** — «что нового» не видно (снижало бы тревогу перед заменой .exe). Показать или убрать.
- **Нет отмены идущей закачки обновления** (`CancellationToken` сервис уже принимает).
- **`ProcessStopFix` без своего денилиста критичных процессов** (полагается на классификацию сканера).

## ⚪ Мелочи (LOW)

- Мёртвые `using` в `MainWindowViewModel.cs` (`System.Collections.Concurrent`, `System.Diagnostics`) после выносов.
- `CleanupAfterUpdate`: при мгновенном перезапуске `.old` залочен старым процессом → удаляется позже (самолечится).
- `Aegis.new.exe`-осиротевший не подчищается при крэше между скачиванием и подменой.
- `ConnectivityWatcher.CheckAsync`: брошенная `connect`-задача при таймауте (unobserved).
- `OnlineReputationChecker.AutoCheckAsync` не принимает `CancellationToken`.
- `StartupProgramRemover` :60 `GetFileNameWithoutExtension(displayName)` усекает имя с точкой — ослабляет fallback-сопоставление.
- `DeleteStartupCompletelyAsync`: `AddProgramsRemoved()` когда снят только автозапуск (не снесена программа) — завышает статистику.
- `UninstallView` тумблер «Спрашивать AI» без `IsVisible="{Binding AiAvailable}"` (в ScansView — с ним).
- ForceDelete закрывает мешающие программы без подтверждения.
- Плитки Дашборда «Удаление программ»/«Удалить файл» — одна иконка `trash`.
- Права администратора только логируются, не гейтят команды (защита — манифест).
