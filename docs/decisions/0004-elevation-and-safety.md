# 0004. Права администратора и безопасность изменений

- Статус: Accepted
- Дата: 2026-06-17

> ПЛАН. Кода ещё нет — фиксируем архитектурное решение до начала разработки.

## Контекст
Aegis правит системные подсистемы (реестр, службы, автозапуск, настройки, драйверы).
Для большинства операций нужны **права администратора**, а сами изменения по своей
природе опасны. Заказчик не разбирается в технологиях и полагается на то, что программа
ничего не сломает безвозвратно. Нужно зафиксировать сквозной **принцип безопасности
изменений** для всего приложения.

## Варианты
1. **`requireAdministrator` в `app.manifest` (элевейшн при старте) + сквозной принцип
   обратимости каждого изменения** (точка восстановления/бэкап ветки ПЕРЕД правкой),
   без безвозвратных удалений, явные предупреждения, полный лог. Один понятный UAC-промпт
   на запуск; единый дисциплинированный путь для всех правок.
2. **Запуск без элевейшна, точечное повышение прав под отдельные операции** (отдельный
   elevated-процесс/брокер по запросу) — меньше прав по умолчанию, но **сложнее
   архитектурно** (IPC с elevated-брокером, множественные UAC-промпты), хуже UX для
   приложения, которое почти всё делает на системном уровне. Отклонён для текущего one-man
   «для себя» сценария (можно вернуться к идее брокера позже, если понадобится).
3. **Удалять/править «как есть» без обязательного бэкапа** (доверять успешности правок) —
   проще и быстрее, но **недопустимо**: нарушает требование обратимости и риск
   безвозвратной поломки. Отклонён.

## Решение
- Приложение запрашивает права администратора через **`app.manifest`
  (`requireAdministrator`)** и проверяет фактические права в рантайме.
- **Любое исправление обратимо:** перед каждой правкой — **точка восстановления и/или
  экспорт затрагиваемой ветки** (см. [0002](0002-backup-rollback.md)).
- **Никаких безвозвратных удалений** без предварительного бэкапа (мусор — в карантин/
  бэкап перед удалением).
- **Явные предупреждения** пользователю перед потенциально рискованными действиями;
  «?»-Explain-тексты простыми словами (суть проблемы + что даст исправление).
- **Полное логирование** критичных операций (Serilog): что, когда, какой бэкап создан,
  результат.
- Работа с реестром/службами — **параметризованно и с обработкой ошибок**; P/Invoke/WMI —
  в изолированных безопасных обёртках.

## Последствия
- **Плюсы:** один понятный UAC-промпт при старте; единый безопасный путь для всех правок;
  любое действие откатываемо; прозрачность через логи и предупреждения; доверие
  заказчика.
- **Минусы / риски:** приложение всегда работает с полными правами админа — повышенная
  ответственность кода (обёртки, валидация, обработка ошибок обязательны); UAC-промпт при
  каждом запуске; обязательный бэкап перед правкой замедляет операции и требует места/
  включённой System Restore.
- **Требует от остального стека:** `app.manifest` с `requireAdministrator` (Фаза 1);
  каждый `Fix` создаёт обратимый бэкап ПЕРЕД правкой (Фазы 2–3); coding-standards под C#
  закрепляют: системное изменение только через обратимую операцию, без безвозвратных
  удалений, requireAdministrator + проверка прав, Serilog на критичных операциях.
