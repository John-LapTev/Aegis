# 0001. Стек: C#/.NET 9 (self-contained .exe)

- Статус: Accepted (выбор UWP-фреймворка **обновлён** в [0006](0006-ui-avalonia.md): UI на **Avalonia**, не WPF)
- Дата: 2026-06-17

> Стек .NET 9 / C# / MVVM / DI / Serilog / xUnit — в силе. Исходно UI планировался на WPF;
> по [ADR 0006](0006-ui-avalonia.md) UI переведён на **Avalonia** (чтобы собирать `.exe` на Linux).
> Остальное (доступ к системе, бэкап/откот, single-file publish) — без изменений.

## Контекст
Aegis — десктоп-приложение под **Windows 10/11** (один пользователь, «для себя»):
оптимизатор + чистильщик + сканер безопасности + тюнер системы. Из задач вытекают
жёсткие требования к стеку:

- **Глубокий доступ к Windows на системном уровне:** реестр, службы, автозапуск,
  процессы, WMI, драйверы, GPU-вендоры (NVIDIA/AMD).
- **Права администратора** (элевейшн через `app.manifest`) — без них половина функций
  недоступна.
- **Самодостаточный дистрибутив:** один `.exe`, который запускается на чистой Win 10/11
  без установки рантайма (single-file self-contained, win-x64).
- **Современный, кастомизируемый UI** с тёмной темой и градиентным акцентом.
- Один разработчик/ассистент, скорость разработки важна.

Заказчик не разбирается в технологиях — стек выбираем мы.

## Варианты
1. **C# / .NET 9 + WPF** — нативный доступ к Windows API (реестр, службы, процессы,
   WMI) через стандартную библиотеку и `System.Management`; зрелый MVVM
   (CommunityToolkit.Mvvm), DI, async; простой single-file self-contained publish
   под win-x64; гибкая кастомная тёмная тема через XAML. Минус: WPF — только Windows
   (для проекта это не минус, цель именно Windows).
2. **WinUI 3 / Windows App SDK** — современнее визуально, но **сложнее упаковка и рантайм**
   для self-contained одиночного `.exe` (зависимости Windows App SDK, нюансы
   unpackaged-деплоя), меньше зрелых примеров системного доступа. Отклонён.
3. **C++ / Qt** — максимальный контроль и производительность, но **дольше разработка**
   (ручной memory management, больше кода под те же задачи), Qt тянет лицензионные/
   дистрибутивные нюансы. Избыточно для one-man проекта. Отклонён.
4. **Rust + Tauri** — лёгкий бинарь и веб-UI, но **слабее прямой доступ** к WMI/реестру/
   службам/драйверам (всё через crates/FFI, больше ручной работы), элевейшн и системная
   интеграция менее «из коробки», экосистема системных Windows-обёрток беднее. Отклонён.
5. **Python + PyInstaller** — быстрый прототип, но **тяжёлый и медленный exe**, слабее
   для системного уровня и элевейшна, проблемы с распространением и антивирус-флагами.
   Отклонён.

## Решение
Берём **C# / .NET 9 + WPF**, публикация — **single-file self-contained, win-x64**.
Архитектура: **MVVM** (CommunityToolkit.Mvvm), **DI**, **async/await** (не блокируем UI).
Логирование — **Serilog**, тесты — **xUnit**. Доступ к системе: `System.Management`
(WMI), `Microsoft.Win32.Registry`, Process/ServiceController API, при необходимости
**P/Invoke** в изолированных безопасных обёртках. GPU: NVIDIA — NVAPI/`nvidia-smi`,
AMD — ADLX/ADL или вендор-реестр (best-effort, поздняя фаза).

## Последствия
- **Плюсы:** прямой и зрелый доступ ко всем нужным Windows-подсистемам; простой
  самодостаточный `.exe` без установки рантайма; быстрая разработка UI и логики;
  богатая экосистема (MVVM/DI/логирование/тесты).
- **Минусы / риски:** привязка к Windows (приемлемо — цель только Windows);
  single-file self-contained `.exe` крупный по размеру (десятки МБ) и может ложно
  триггерить антивирусы — учесть при упаковке (Фаза 7); WPF-стилизация тёмной темы
  требует ручной работы по токенам.
- **Требует от остального стека:** `app.manifest` с `requireAdministrator` (см.
  [0004](0004-elevation-and-safety.md)); coding-standards под C# 13/.NET 9 (nullable
  enable, без warnings, P/Invoke/WMI в обёртках); движок сканов на общем интерфейсе
  `IScanner` → находки `Finding` (severity + Explain-текст для «?»).
