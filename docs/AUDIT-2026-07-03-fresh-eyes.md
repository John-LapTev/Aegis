## 🔴 Важное (чинить в первую очередь)

### 1. [HIGH] Дедуп версий: сопоставление слепо к разрядности x86/x64 → помечается НЕ тот пакет
**Подтверждено двумя независимыми аудиторами (промпты ИИ + логика).**
`DashboardViewModel.cs` `StripVersionSuffix` режет имя по ПЕРВОМУ `" ("`. У имён VC++ разрядность стоит в скобках перед версией: `…Redistributable (x64) - 14.34…` → срезается и `(x64)`, и версия. Дальше `NamesMatch` по двустороннему `StartsWith` + `FirstOrDefault` совпадает и с x64, и с x86 и берёт первый по алфавиту (x64 раньше x86). Итог: ИИ по своему же промпту помечает к удалению устаревший **x86**, а в список удаления попадает **x64** — тот, который надо было ОСТАВИТЬ. Весь аккуратный промпт «x86≠x64» обесценивается на слое сопоставления. Тестов на матчинг нет вообще.
**Фикс:** сопоставлять по ПОЛНОМУ имени с сохранением `(x86)/(x64)`; при неоднозначности (совпало >1) — строку DEL пропускать (fail-safe: лучше не пометить, чем пометить не то). Покрыть тестом: пара x64/x86, DEL на x86 → в дубли попадает именно x86.

### 2. [HIGH] Чистка остатков запускается, даже когда программа фактически НЕ удалена → риск снести живую папку/данные
`DashboardViewModel.UninstallAsync`: при `StillRegistered==true` (деинсталлятор вернул успех, но программа на месте — ждёт ребута / это лаунчер/игра) код всё равно идёт в `_leftovers.ScanAsync`→`RemoveAsync`, а тот удаляет папки **насовсем**. `LeftoverService.Scan` добавляет ещё существующую папку установки и папки в AppData по имени. Можно безвозвратно снести папку установки и данные ЖИВОЙ программы, при этом окно остатков жёстко пишет «Программа удалена» — ложь в этой ветке.
Смежно: [MEDIUM] у асинхронных деинсталляторов (InnoSetup/NSIS, перезапускают себя из temp) `reg query` сразу после выхода родителя даёт ложное `StillRegistered=true` → преждевременное предложение удалить папку, пока настоящий деинсталлятор ещё в ней работает (гонка).
**Фикс:** при `StillRegistered` не предлагать безвозвратное удаление (минимум Корзина) и не писать «удалена»; перед вердиктом — 2–3 повторных опроса ветки с паузой.

### 3. [HIGH] Файлы/папки-остатки удаляются безвозвратно, без бэкапа
`LeftoverService.Remove` — `Directory.Delete(recursive:true)`/`File.Delete` напрямую (не Корзина, не карантин, отката нет). Бэкап есть только у реестра. Нарушает обещание «всё обратимо». Это осознанное решение (правка Ивана 1204: большие папки не влезали в Корзину) — но undo нет. Также PathSafety на удалении файлов/папок повторно НЕ перепроверяется (для реестра — перепроверяется).
**Фикс (на решение Ивана):** удалять остатки в Корзину, «навсегда» — только явный fallback с отдельным подтверждением; либо через `QuarantineStore` (уже есть в проекте). Перепроверять PathSafety прямо в `Remove`.

### 4. [HIGH] Ссылки из свободного ответа ИИ не проверяются и лживо подписаны «официальная страница»
`FindingViewModel` (`OnAiAnswerChanged`/`LinkLabel`): любой URL из текста модели рендерится кнопкой с подписью **всегда** «Открыть официальную страницу (host)» для любого хоста, без allowlist. Модель галлюцинирует URL; в контексте «скачать драйвер» доверчивый пользователь кликнет на фейковый `realtek-drivers[.]com` с ярлыком «официальная». (В `DeviceUpdateLookup` allowlist доменов есть — здесь нет.)
**Фикс:** прогонять URL из ответа ИИ через тот же `TrustedDomains`-allowlist; неофициальные — не давать кнопкой; никогда не писать «официальная», если домен не подтверждён (нейтральное «Открыть ссылку (host)»).

### 5. [HIGH] Окно «Удаление файла»: «Удалить навсегда» неотличимо от «Отменить»
`DeleteConfirmWindow.axaml`: «Отменить» и «Удалить навсегда» — оба серые `secondary`; безвозвратное действие выглядит так же безобидно, как отмена. Для «нулевой» аудитории — прямой риск случайного безвозвратного удаления.
**Фикс:** «Удалить навсегда» — в опасный стиль (красная окантовка/текст), безопасную «В корзину» оставить `primary`.

## 🟡 Стоит поправить (MEDIUM)

- **Остатки-реестр по ИЗДАТЕЛЮ сносят общую ветку многопродуктового вендора.** `LeftoverService.RegistryCandidates` добавляет `SOFTWARE\<Publisher>`; PathSafety блокирует только зашитый список (microsoft/google/valve…), но не «средних» вендоров. Пример: удаляем CCleaner (издатель Piriform) → предлагается вся `HKCU\SOFTWARE\Piriform`, вместе с настройками Recuva. Для папок издателя намеренно НЕ используют — для реестра осталось. **Фикс:** не использовать издателя как самостоятельный ключ ветки (структурное правило: только если под веткой один продукт).
- **Невалидная InstallDate роняет ВСЮ запись — программа исчезает из списка.** `InstalledProgramsProbe.ReadInstallDate` проверяет `day<=31`, но не «день в месяце»: `InstallDate="20230231"` → `new DateOnly(2023,2,31)` бросает → `catch` в `ReadHive` пропускает запись целиком → программу нельзя увидеть/удалить. **Фикс:** `TryParseExact("yyyyMMdd")` / `DaysInMonth`; опциональное поле не должно бросать вверх.
- **Жаргон для аудитории «не разбирается в ПК».** «VirusTotal», «деинсталлятор», «реестр», «остатки», «распространяемые пакеты», «рантайм», «разрядность x86/x64» выдаются прямо пользователю. **Фикс:** бытовые формулировки или прятать термин в «?»-подсказку.
- **4 раздела без видимой кнопки «Назад».** «Удаление программ», «Оптимизация», «Сравнить состояние», «Удалить файл» доступны только плиткой Дашборда; выйти — Escape (нигде не подписан) или клик по «Дашборд». Новичок может застрять. **Фикс:** явная кнопка «← К Дашборду» в шапке этих разделов.
- **«Мёртвый клик» «Проверить этот раздел» во время другого скана.** Кнопка активна по `!SelectedGroup.IsScanning`, но `ScanGroupAsync` выходит по глобальному `IsScanning` → клик молча ничего не делает. **Фикс:** блокировать по глобальному `IsScanning` или подсказка «дождитесь окончания».
- **Кнопка «Дубли версий (AI)» работает в обход тумблера «Спрашивать AI».** Видна по `CanCheckVersions` (=IsConfigured), минуя `AiEnabled`. *Замечание: это осознанное решение Ивана (1214 — «не прятать, иначе не найти»). Оставляю как есть, если Иван не передумал; можно повесить подпись «использует AI».* Плюс в разделе два тумблера одного состояния (шапка убрана в 2.74.0 — проверить, не осталось ли дубля).
- **Промпт дедупа: не разграничены .NET Framework и .NET (Core/5+).** Маркер `.net` захватывает `.NET Framework 3.5/4.8` — отдельный критичный компонент Windows, который НЕ заменяется новым .NET. **Фикс:** явное правило в промпт «.NET Framework 2.0/3.5/4.x — не помечать никогда».
- **Косвенная prompt-injection из веб-сниппетов.** `WebAugmentedAiAssistant` вставляет заголовки/сниппеты с «ОБЯЗАТЕЛЬНО опирайся», без пометки недоверенности. Ущерб ограничен (эти пути только показывают текст), но для доверчивой аудитории плохой совет вреден. **Фикс:** обрамлять как непроверенное, в связке с allowlist ссылок (#4).
- **`RebootRollbackScheduler.Clear` глотает исключения без лога.** Пустой `catch` при удалении pending-файла/RunOnce → устаревший pending может повторно запросить откат после следующей перезагрузки. **Фикс:** `Log.Warning`.

## ⚪ Мелочи (LOW)

- Магические строки ключей `Data`-словаря находок (`"kind"`, `"section"`, `"paths"`, `"hive"`…) — производитель и потребитель договариваются нетипизированными строками; опечатка молча ломает починку. **Фикс:** завести `FindingDataKeys` рядом с `FindingKinds`.
- Дедуп программ по `DisplayName` — keep-first: два разных продукта с одним именем → второй не показывается, `UninstallCommand` от первого.
- Счётчик выбранных и масс-удаление считают только `VisiblePrograms`: отметил 5 → сузил поиск → удалятся видимые, часть выбора «спрячется».
- Автозапуск-остатки: `Contains` вместо `StartsWith` по пути установки — путь как аргумент чужой команды даст ложный флаг.
- Дублирование: `Process.Start`-обёртки (`OpenItem`/`OpenPath`/`OpenUrl`), парсинг exe (`ScanViewHelpers.ExtractExecutablePath` vs `FindingViewModel.ParseExecutable`), synthetic-`Finding`.
- Мёртвые/дублирующие члены `FindingViewModel`: `IsRawDisk`≡`IsFilesystemUnreadable`, `IsDiskCard` не используется; осиротевший doc-комментарий в хвосте `MainWindowViewModel.cs`.
- Разнобой слов: «Выделить всё/все», «Снять/Сбросить выделение»; закрытие ответа AI — «✕» vs «Скрыть».
- Захардкожена «Норма до 85°» в карточке стресс-теста.
- Дрейф доков: `CLAUDE.md` (v2.4) пишет «Дашборд убран», но он — первый пункт рельса.
- CancellationToken не проброшен в конвейер сканов (отмена готова в коде, но VM зовёт без токена) — сканы нельзя прервать. Если отмена не нужна продуктово — оставить.

## Рефакторинг (ответ на прямой вопрос Ивана)
**Критичных дефектов нет, немедленного «большого» рефакторинга не требуется** — можно продолжать разработку. Техдолг умеренный и локализован в UI-слое: «горячая четвёрка» за hard-лимитами проекта (сервис/VM ~500):
